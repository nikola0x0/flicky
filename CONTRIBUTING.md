# Contributing to Flicky

Developer setup, local commands, and conventions. For the product overview and
what Flicky *is*, see [`README.md`](README.md).

## Prerequisites

Requires [Bun](https://bun.sh) ≥ 1.3 and the [Sui CLI](https://docs.sui.io/guides/developer/getting-started/sui-install)
(for `sui move build` / `test` / `publish`). Always use `bun` (≥ 1.3), never
`npm` / `pnpm` / `yarn`. The workspace is pinned to `bun@1.3.9`.

## Stack layout

Bun workspaces + Turborepo monorepo.

```
flicky/
├── apps/
│   ├── web        # Vite + React 19 + Tailwind v4 + shadcn/ui — swipe UI, lockup view, share card
│   ├── server     # Bun runtime — WebSocket relay, indexer, settled-redeem keeper, sponsored-gas, AI Deckmaster
│   └── contracts  # Move package + TS deploy/upgrade/codegen scripts
└── packages/
    └── ui         # @workspace/ui — shared shadcn/ui components, hooks, lib utils, globals.css
```

## Getting started

```bash
bun install                          # install all workspace deps
bun dev                              # turbo dev — runs web + server in parallel
```

Individual apps:

```bash
bun --filter web dev                 # Vite dev server (default :5173)
bun --filter server dev              # Bun --hot server (default :3001)
bun --filter @flicky/contracts test  # sui move test (Move unit tests)
```

### First-time developer onboarding

1. `bun install`
2. `cd apps/contracts && cp .env.example .env.local && cp .env.example .env` — fill
   `SUI_DEPLOYER_PRIVATE_KEY` only if you intend to publish/upgrade. For read-only
   dev, you can skip it.
3. `bun --filter @flicky/contracts codegen` — generates `apps/web/src/sui/gen/`
   against the currently-deployed package id from `deployed.json`. Required before
   `bun --filter web dev` typechecks.
4. `bun dev` to spin up web + server.

## Working with the Move package

Run these inside `apps/contracts/` (first-time setup or after any Move change):

```bash
bun run test                         # sui move test --gas-limit 100000000000
bun run build                        # sui move build
bun run publish                      # first deploy to testnet (writes deployed.json)
bun run upgrade                      # subsequent upgrades (preserves originalPackageId)
bun run codegen                      # regenerate TS bindings → apps/web/src/sui/gen/
```

Once published, `apps/web/src/sui/gen/` holds typed `moveCall` builders for both the
flicky package and DeepBook Predict. **The directory is gitignored — every developer
regenerates locally after `bun install`.**

The current testnet deployment lives in [`apps/contracts/deployed.json`](apps/contracts/deployed.json)
— that file is the source of truth, regenerated by `bun --filter @flicky/contracts publish` / `upgrade`.

## Other workspace tasks

```bash
bun typecheck      # turbo typecheck across all workspaces (tsc --noEmit per workspace)
bun test           # bun-test in each app (server + web) — see each app README
bun lint           # turbo lint (eslint per workspace)
bun format         # prettier write
bun build          # turbo build
```

## Adding shadcn components

Components live in `packages/ui/src/components` and are imported via
`@workspace/ui/components/<name>`. To add one:

```bash
bunx --bun shadcn@latest add <component> -c apps/web
```

The shadcn CLI is configured in `apps/web/components.json` to install into the shared
`@workspace/ui` package (not into the web app's local `src/components`) and to use the
shared `packages/ui/src/styles/globals.css` Tailwind stylesheet. The `radix-luma` style
and `neutral` base color are fixed — don't change them ad hoc.

Web-app-local components go in `apps/web/src/components` and are imported via the
`@/components/...` alias (configured in `apps/web/vite.config.ts`).

## Code style

- **Prettier** — no semicolons, double quotes, 2-space, trailing comma `es5`, print
  width 80. The Tailwind plugin runs `cn` / `cva` through class-sorting against
  `packages/ui/src/styles/globals.css`.
- **TypeScript** — `strict: true` everywhere; `moduleResolution: bundler`, `target: ES2022`.
- **Server** is ESM (`"type": "module"`) and runs directly via `bun --hot src/index.ts` —
  no bundling in dev. Prefer `Bun.*` APIs over Node shims.
- **ESLint** configs are per-workspace flat configs. `packages/ui` intentionally omits
  `eslint-plugin-react-refresh` (it's a library, not a Vite app) — don't add it back when
  wiring new shadcn components.

## Architectural constraints (load-bearing — preserve these)

- **Player-signed swipe PTBs are atomic.** Each swipe is a single PTB that calls
  `predict::mint` on the player's own `PredictManager` and `duel::record_swipe` on the
  shared `Duel` in the same transaction. Forced by Predict's `sender == manager.owner()`
  invariant — don't route mint through the keeper.
- **The `Duel` shared object does NOT hold Predict positions.** Each player owns their
  `PredictManager`; the `Duel` escrows the dUSDC side-pot and records swipes. Settlement
  reads scores from `record_swipe` / `settle_card` data, not from positions.
- **Two tiers share one engine.** The free/practice tier runs the same swipe + lockup +
  settle flow as staked (`*_free` entrypoints), only with `predict::mint` and the dUSDC
  stake removed. Gate the money flow; keep the engine.
- **Sponsored gas end-to-end.** Player zkLogin wallets only ever hold dUSDC. Every new PTB
  the player signs goes through the sponsored-gas service in `apps/server`.
- **Commit-reveal deck.** Cards are hashed at duel creation, revealed at match start.
  Never expose the unrevealed deck through the WebSocket relay before reveal time.
- **`p_swiped` is snapshotted at swipe time.** Scoring uses Predict's SVI surface read
  inside the swipe PTB, not a later read.
